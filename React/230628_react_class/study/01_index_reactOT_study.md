## 리액트는 '라이브러리 인가, 프레임워크 인가'


### 라이브러리 VS 프레임워크

```
a) '폴더 구조' 가 있는지 여부 
    - 라이브러리는 폴더 구조가 없음. 
    - 프레임워크는 폴더 구조가 있음. 

b) 리액트를 프레임워크라고 볼 수도 있는 이유 
    - 리액트 개발할 때, 편하게, 초기 셋팅을 받아서 사용할 수 있게 해주기 때문에, 프레임워크라고 보는 블로거도 있음.
    ex) npx create-react-app [폴더명] 

c) 리액트를 '라이브러리' 로 봐야 하는 이유 
    - 공식홈페이지에서 라이브러리 라고 말함 (https://ko.legacy.reactjs.org/)

```


## 리액트 특징 

```
1. '자바스크립트' 를 통해서 'HTML 을 조작' 할 수 있음. 
    - 이 기능을 원래 수행하던게 document.creatElement 같이, DOM 요소 조작 이었음. 
    - 이 부분에 보완이 들어가게 됨. 

2. '가상의 DOM'과 비교해서, 변경된 부분만 다시 그림 (update)
    - '가상의 DOM' 은 
        1) 메모리에 남아있는 상태 
        2) 실제 페이지의 DOM 은 분리 되어 있음. 
        3) react 의 상태가 변하면, 가상 DOM 이 생성되고, 이전의 가상 DOM 과 비교해서, '틀린 그림' 을 찾고 업데이트 한다. 

    - 이렇게 변화된 부분만 그리기 때문에 
        1) 효율적이다. 

3. 서버 사이드 렌더링과의 비교 
    - PHP EJS 를 사용해서, 서버에서 페이지를 만드는 경우 | ❓❓ 서버 사이드 렌더링의 한계점이 뭘까? 추가 필요 📛📛 
    - 이렇게 하다가, 클라이언트에서 페이지를 만들어야할 필요가 생김 
    - 그래서, 클라이언트 사이드 렌더링이 대두됨. (react, vue, 앵귤러 등이 등장)

4. 기타 
    - DOM 조작을 쉽게 하기 위해 사용
    - single page application 을 만들기 위해 사용 
    - 새로 고침이 없음. | 잘 이해가 안 됨, 보완 필요 | ❓❓ 
        1) url 은 변함. 변하는 경로를 react-router 가 잡아준다. 

        2) 내용물을 바꾼다. 그래서, 페이지가 바뀐 것 처럼! 속인다. 

```
